    # Gitのコマンドについて

    # 一部分だけコミットする

    # git add -p

    一部分だけコミットしたい時ってあるよね

    そんな時は次のコマンドを使うと良いよ

    $ git add -p {ファイル}

    このコマンドの後に次の質問が来るよ

    Stage this hunk [y,n,q,a,d,/,e,?]

    ブロックごとにaddするか質問をしているよ

    # commitログの操作

    # 1.過去のcommitの修正

    # 1-1.直前のcommitの場合

    commitに --amendオプションをつけると、直前のcommitの内容を上書きする形で修正できるよ

    $ git commit --amend
    # 用途

    直前のcommitを直す

    • addしたCommitした内容を上書きするため使うよ
    • Commit Commentを修正するために使うよ

    # 1-2.過去のcommitを修正する場合

    修正したい過去のcommitより後のcommitを指定すると、HEADから修正したいcommitまでをどう修正するか一覧が表示されるよ

    commitの場所は大体HEAD~~みたいに使うことが多いかな

    $ git rebase -i <commit>

    上のCommandを実行すると下のようなmessageの編集画面になるよ

    pick 8145f1c Fix screen rotation problem
pick d90db4a Add new CSS
pick 646bf79 Fix screen rotation problem
pick 71a6940 Replace submit function

# Rebase ed7420a..71a6940 onto ed7420a
#
# Commands:
#  p, pick = use commit
#  r, reword = use commit, but edit the commit message
#  e, edit = use commit, but stop for amending
#  s, squash = use commit, but meld into previous commit
#  f, fixup = like "squash", but discard this commit's log message
#  x, exec = run command (the rest of the line) using shell
#
# These lines can be re-ordered; they are executed from top to bottom.
#
# If you remove a line here THAT COMMIT WILL BE LOST.
# However, if you remove everything, the rebase will be aborted.
#

    上を見ると全てpickになっているので、必要な

    また、この時に行を入れ替えるとcommitの順番を入れ替えることができるよ

    • pickの場合は何もしない
    • rewardはcommit messageを修正する
    • editはcommitのところで一時停止
      • この時にshellに戻されるので、commitの追加やamendで上書き修正ができるよ ex）
    <commitの内容の修正>
$ git commit --amend
$ git rebase --continue

    editにしたcommitの位置になったら、編集できるようになるよ
    やりたいことが終わったらgit rebase --continueで次のrebaseを続けるよ

    • squashはcommitをまとめるよ
      • この時にcommit commentを修正できるよ
    • fixupはcommit commentの修正をしないけどcommitをまとめるよ
    • execはcommitに対して何かするわけじゃないけど、commitの位置で特定の処理をできるよ

    ex)

    pick 8145f1c Fix screen rotation problem
exec make test
pick d90db4a Add new CSS
pick 646bf79 Fix screen rotation problem
exec make test
pick 71a6940 Replace submit function
    # 用途
    • 過去のcommitの内容を上書きするとき
    • 過去のcommitの結合したいとき
    • 過去のcommitの順番を変えたいとき
    • 過去のcommitログを変えたいとき

    WARNING

    リモートリポジトリにpushした内容でrebaseを行わないこと！

    git push --forceをしないといけなくなるよ！

    # おまけ
    # rebaseをやめたくなった時

    --abortをつけるとrebaseを終わらせられるよ

    $ git rebase --abort
    # 間違えた時

    pushしていなければ、下のcommand実行で戻せるよ

    $ git reset --hard ORIG_HEAD

    TIP

    reflogのpointを使っても元に戻せるよ

    # 2.無かったことにする

    # 2-1.直前のcommitを無かったことにする

    $ git reset --herd HEAD~

    # 2-2.直前のresetを無かったことにする

    $ git reset --hard ORIG_HEAD
    # ORIG_HEADってなに？

    HEADが移動するcommandを叩いた際に、command実行時のHEADの位置のこと
    git reflogを使うと見れるよ(3で紹介するよ)

    # 3.Reflogについて

    # 3-1.HEADの移動履歴をみる

    $ git reflog

    reflogを実行すると、過去にHEADが指していたcommitの一覧が見れるよ

    e6e1d8a0f (HEAD -> feature/ten_commandments) HEAD@{0}: commit (amend): feat:レスポンシブ対応
c5b7259b9 (origin/feature/ten_commandments) HEAD@{1}: commit: feat:レスポンシブ対応
c2228d76f HEAD@{2}: commit: feat:トップページを二つに割る
757f60a1a (origin/master, origin/HEAD, master) HEAD@{3}: checkout: moving from master to feature/ten_commandments
757f60a1a (origin/master, origin/HEAD, master) HEAD@{4}: pull: Fast-forward

    # 3-2.rebaseを無かったことにする

    $ git reset --hard {commit-reflog}

    reflog実行時に表示されるハッシュかHEAD@{数字}を指定して実行するよ

    HEADの操作履歴のcommitを元に戻しているよ

    # 用途
    • 大体のことをやり直したいとき

    # 4.branchを間違えた時

    # 4-1.branchを間違えてcommitした場合

    cherry-pickをするといいよ

    移動させたい先のbranchに移動して、次のcommandを実行すればいいよ

    $ git cherry-pick "{commit}"

    # 5.分類できなかったおまけ

    # 5-1.特定のコメントを含むcommitを探したい

    --grepオプションを使うといいよ

    $ git log --grep "{pattern}"
